# 알밤마켓 관리자(Admin) PRD

> 출처: [`PRD.md`](./PRD.md)(F001~F021 사용자 기능) · [`ROADMAP.md`](./ROADMAP.md)(Phase 7 T073/T074) · [`ISSUES.md`](./ISSUES.md)(확정 정책 상수) 기반. 본 문서는 **운영자(관리자) 관점의 신규 범위**를 정의한다.
> 작성일: 2026-07-04 · 상태: **요구사항 정의(착수 전)**. 미결정 항목은 문서 말미 [OPEN](#-확인-필요--미결정-open)을 참조.

> **🔀 분리 방침 확정 (2026-07-04)**: 관리자 콘솔은 일반 사이트와 **분리된 별도 앱/배포**로 구축한다(같은 앱 `app/admin/**` 라우트 그룹 전제였던 아래 서술은 이 방침으로 대체됨). 목적: **보안 격리·독립 배포·운영 편의**. 이에 따라 ① `SUPABASE_SERVICE_ROLE_KEY`는 **admin 앱 env에만** 주입(일반 사이트 미노출), ② admin 앱은 **자체 미들웨어(`proxy.ts`) 가드**를 가지며, ③ Supabase 백엔드(`admin_users`/`reports`/RLS/admin RPC)는 **양 앱이 공유**한다(백엔드는 분리 대상 아님).
> **OPEN-A(분리 형태)**: **모노레포 2앱(`packages/shared` 공유, 권장)** vs **완전 별도 레포(submodule/private 패키지로 공유층 동기화)** — 착수 전 확정 필요. 전체 구현 로드맵 재편은 이 선택에 종속(`ROADMAP_ADMIN.md` 참조).

## 🎯 핵심 정보

**목적**: MVP(Phase 0~6)로 완성된 C2C 경매 직거래 플랫폼을 안전하게 운영하기 위한 **관리자 콘솔**을 구축한다. 회원·상품·거래·신고를 모니터링하고, 제재/강제 조치로 어뷰징과 분쟁에 대응하며, 정책 수치를 운영 중 조정한다.
**대상 사용자**: 서비스 운영자(관리자). 일반 회원(판매자/구매자)과 달리 **전 회원·전 거래·전 콘텐츠에 대한 감독 권한**을 가진다. MVP 단계에서는 소수 운영자(1인 개발자 포함)를 상정한다.
**로드맵상 위치**: Phase 7 미착수 항목 **T074**(관리자/신고·제재) + **T073**(판매/구매 통계·패널티 현황 대시보드). 본 PRD가 두 Task의 상세 요구사항을 확정한다.

### 왜 지금 필요한가

- 어뷰징(허위 매물, 악성 입찰, 노쇼 반복, 욕설/개인정보 유출 채팅)에 대응할 **운영 수단이 전무**하다.
- 현재 유일한 특권 인프라는 `lib/supabase/admin.ts`의 `createAdminClient()`(service_role, RLS 우회, 서버 전용)뿐이며, **RBAC 개념·관리자 판별·신고 접수 경로가 모두 부재**하다.
- 분쟁 거래(강제 취소/완료), 정책 위반 회원 제재, 정책값 실시간 조정을 사람이 DB에 직접 손대지 않고 처리할 UI가 필요하다.

## 🚦 개발 원칙 (프로젝트 공통 준수)

관리자 페이지도 프로젝트 표준 개발 원칙을 그대로 따른다.

1. **화면 우선 개발(Mock First)**: DB/권한 연결 전에 Mock 데이터 + 공용 타입으로 관리자 화면을 먼저 완성한다.
2. **단일 타입 계약**: 관리자용 신규 타입(`lib/types/admin*`)도 Mock·실DB가 동일 계약을 공유. 실데이터 전환 시 조회부만 교체.
3. **`/sample` 전시**: 관리자 컴포넌트도 `/sample`에 전시(단, 권한 게이트 뒤의 실제 화면은 `/admin`).
4. **snake↔camel 매핑**: 신규 테이블도 `lib/queries/_map.ts` 패턴으로 매핑.
5. **스키마 변경 절차**: 원격 Supabase(`zmeyfvfkqnemnzafpzmn`)에 `apply_migration` 후 `generate_typescript_types`로 `lib/database.types.ts` 재생성(로컬 SQL 파일 없음).

## 🔐 선행 과제 — 관리자 권한 모델

> **모든 관리자 기능(FA010~FA090)의 전제.** 이 절이 완료되어야 나머지 기능을 안전하게 노출할 수 있다.

### 현재 상태(근거)

- `profiles`에 `role`/`is_admin` 컬럼 **없음**. RBAC 개념 부재. 코드의 "role"은 거래 맥락(seller/buyer)과 `ratings.role`(as_seller/as_buyer)뿐이다.
- 유일한 특권 인프라: `lib/supabase/admin.ts`의 `createAdminClient()`(service_role, RLS 우회, **서버 전용**).
- 제재 토대: `penalties` 테이블(user_id, reason, penalty_type). **신고(reports) 테이블은 없음 → 신규 설계 필요.**

### 결정된 권한 판별 방식

| 항목              | 결정                                                                                     | 비고                                                                           |
| ----------------- | ---------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| 관리자 판별       | **별도 `admin_users` 테이블**(profiles에 컬럼 추가하지 않음)                             | user_id PK/FK → `profiles.id`                                                  |
| 관리자 등급       | **단일 권한으로 시작**, `admin_users.role` 컬럼으로 **다단계 확장 여지 확보**            | 지금은 모든 관리자 동일 권한. 세분화 시점은 [OPEN-5](#-확인-필요--미결정-open) |
| service_role 사용 | 강제/제재 등 RLS 우회가 필요한 mutation에 한해 **서버 전용**(`createAdminClient`) 재사용 | 클라이언트 노출 절대 금지                                                      |
| 감사 로그         | **모든 관리자 조치를 `admin_action_logs`에 기록**(필수)                                  | 채택 여부는 [OPEN-3](#-확인-필요--미결정-open)에 병기                          |

### 접근 통제 계층 (3중 가드)

| 계층                               | 대상                           | 판정 방식                                                                                                                                   |
| ---------------------------------- | ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **미들웨어 가드**(루트 `proxy.ts`) | `/admin/**` 라우트 진입        | 세션 유무 + `admin_users` 소속 여부. 비관리자는 홈(또는 404)으로 리다이렉트. `createServerClient`~`getClaims` 사이 코드 삽입 금지 규칙 준수 |
| **RLS 정책**                       | 관리자 전용 테이블/조회        | `admin_users`에 `auth.uid()`가 존재할 때만 허용하는 정책(예: `is_admin()` SQL 헬퍼)                                                         |
| **RPC 권한 검증**                  | admin 전용 mutation(강제 조치) | 각 admin RPC 내부에서 `admin_users` 확인 후 실행. 위반 시 예외                                                                              |

- 참고 라우트 모델: `app/my-products/page.tsx`(로그인 게이트 + 필터 + 서버 쿼리). 관리자 페이지는 이 위에 **관리자 게이트**를 추가한 형태.
- 라우트 그룹: `app/admin/**`(레이아웃에서 관리자 게이트 + 사이드바 네비 제공).

### 최초 관리자 지정(부트스트랩)

- `admin_users`에 UI로 관리자를 추가하려면 이미 관리자가 있어야 하는 순환 문제가 있다.
- **최초 1명은 수동 부여**: Supabase MCP `execute_sql`(또는 대시보드)로 대상 `profiles.id`를 `admin_users`에 직접 INSERT(`granted_by=NULL` 허용, `role='admin'`). 이후 관리자는 콘솔의 FA023(관리자 지정)로 추가/회수한다.

```
🛡️ 알밤마켓 관리자 콘솔  [/admin/**]  (admin_users 소속만 접근)
```

## 🔄 관리자 조치 → 상태 전이 매핑

> 기존 사용자 도메인 상태값/RPC를 재사용하되, 관리자 강제 실행 경로가 필요한 곳은 **admin RPC 신규**로 위임(권한 검증·원자성·감사 로그 적재 일체화).

### 상품(ProductStatus) 관련

```text
active ─▶ withdrawn   # FA031 강제 내림 (판매자 의사 무관, 관리자 사유 기록)
active ─▶ won/failed  # FA032 경매 강제 종료 (기존 close 로직 재사용, 즉시 판정)
(any)  ─▶ is_blinded  # FA031 블라인드 (상태값 불변 + 숨김 플래그)
```

### 거래(TransactionStatus) 관련

```text
pending ─▶ canceled    # FA041 분쟁 강제 취소 (연쇄 이양 없이 거래 종료)
pending ─▶ completed   # FA041 분쟁 강제 완료 (평판 반영 여부는 정책 확인)
```

### 회원/평판 관련

```text
회원        ─▶ 정지(user_suspensions/컬럼)   # FA022 (미들웨어/RLS에서 차단 반영)
penalties   ─▶ insert/delete                 # FA024 (ISSUE-004 30일 3회 누적 연동)
profiles    ─▶ nickname 강제 변경            # FA025
ratings     ─▶ 삭제/블라인드 → 평판 재계산   # FA080 (calc_reputation_level 재호출)
```

### 신규 admin RPC 후보 (service_role/권한 검증 내장, 감사 로그 적재)

| RPC 후보                                                                                            | 역할                   | 대응 기능         |
| --------------------------------------------------------------------------------------------------- | ---------------------- | ----------------- |
| `admin_suspend_user(user_id, reason, ends_at)`                                                      | 계정 정지 부여         | FA022             |
| `admin_lift_suspension(user_id)`                                                                    | 정지 해제              | FA022             |
| `admin_grant_penalty(user_id, reason, type)` / `admin_revoke_penalty(id)`                           | 패널티 수동 부여/해제  | FA024             |
| `admin_force_withdraw_product(product_id, reason)`                                                  | 상품 강제 내림         | FA031             |
| `admin_blind_content(target_type, target_id, reason)`                                               | 콘텐츠 블라인드        | FA031/FA070/FA080 |
| `admin_force_close_auction(product_id, reason)`                                                     | 경매 강제 종료         | FA032             |
| `admin_force_cancel_transaction(tx_id, reason)` / `admin_force_complete_transaction(tx_id, reason)` | 분쟁 처리              | FA041             |
| `admin_resolve_report(report_id, resolution)`                                                       | 신고 처리              | FA051             |

## 🔒 권한/보안 (RLS · NFR)

| 분류               | 요구사항                                                                                                                                                           |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 관리자 판별        | `admin_users` 소속 여부로 판별. `is_admin()` SQL 헬퍼(`auth.uid()` in admin_users). `profiles`에 role 컬럼 추가하지 않음                                           |
| RLS                | 신규 테이블(admin_users/reports/admin_action_logs/user_suspensions)은 **관리자만 조회/수정**. `reports`는 예외적으로 **신고자 본인 insert 허용**(FA050)            |
| service_role 범위  | RLS 우회가 불가피한 강제 조치 mutation에 한해 **서버 전용** `createAdminClient` 사용. 클라이언트 번들 노출 금지                                                    |
| 미들웨어 가드      | `/admin/**`는 세션 + 관리자 검증. `createServerClient`~`getClaims` 사이 코드 삽입 금지, `supabaseResponse` 반환 규칙(CLAUDE.md) 준수                               |
| 감사 로그          | **모든 관리자 조치는 `admin_action_logs`에 필수 기록**(누가·언제·무엇을·왜). 사후 추적/책임성 확보                                                                 |
| 채팅/개인정보 접근 | 관리자의 메시지 열람은 **신고 접수된 방/메시지로 범위 제한** 권장(전체 열람 지양). 열람도 감사 로그 대상. 최소열람 원칙. 수준은 [OPEN-6](#-확인-필요--미결정-open) |
| 정책 무결성        | FA061 정책 조정은 기존 RPC 클램프(예: 자동완료 24~168h, ISSUE-002)와 모순되지 않도록 **범위 검증** 후 저장                                                         |
| advisor            | 신규 테이블 RLS 활성화·FK 커버링 인덱스(ISSUE-019 패턴)로 security/performance advisor ERROR 0 유지                                                                |

### 기존 확정 정책과의 정합성 (모순 금지)

- 패널티 이용 제한: **최근 30일 3회 누적 시 경매 등록 차단**(ISSUE-004, `penalty_restriction_threshold=3`/`penalty_window_days=30`). FA024 수동 부여도 이 누적에 합산된다.
- 자동완료 대기: **기본 24h**(ISSUE-002, `auto_complete_wait_hours`, RPC 클램프 24~168h). FA042/FA061과 정합.
- 자동 종료/완료: **pg_cron + DB 함수**(ISSUE-008). FA013 시스템 상태가 이 잡을 모니터링.
- 입찰 증가폭 1,000원(ISSUE-003), 연쇄 이양 즉시(ISSUE-007) 등 확정값 유지.
